每个 Bean 在初始化时确实会被所有 BPP 依次处理，但 BPP 不会“反射扫注解”，而是读取 Spring 预先缓存好的这个类service的注解元数据。
每个 Bean 都会依次经过所有 BeanPostProcessor，BPP读取注解元数据，决定当前类需不需要本BPP增强：
TransactionalAnnotationBeanPostProcessor 读取注解元数据，发现@Transaction注解 → 注册TransactionAdvisor
AsyncAnnotationBeanPostProcessor → AsyncAdvisor
MethodValidationPostProcessor → MethodValidationAdvisor
它们只是把 Advisor 注册到 Bean 的 metadata 中。

Advisor = Pointcut（切点，简单理解为哪些函数需要增强，即哪些函数有@Transaction注解） 
+ Advice（拦截器，怎么增强，函数执行前后增加什么处理，这个每种Advisor自己做）

③ AutoProxyCreator 创建一个代理对象（JDK/CGLIB）
永远只创建 一个代理对象：
UserService$$EnhancerBySpringCGLIB$$xxxx
不会出现代理套代理。

Bean 初始化
    ↓  所有 BPP 读取注解元数据（缓存）
BPP1：发现 @Transactional → 贡献 TransactionAdvisor
BPP2：发现 @Async → 贡献 AsyncAdvisor
BPP3：发现 @Validated → 贡献 MethodValidationAdvisor
    ↓
AutoProxyCreator 收集所有 Advisor
    ↓
创建一个代理对象，代理对象，整个对象保存/缓存一个List<Advisor>，Advisor = Pointcut + Advice
    ↓
方法调用时，第一次时扫一遍List<Advisor>，
根据Pointcut匹配与否，决定把对应的Advice转化为MethodInterceptor，加入到属于这个方法的的List<MethodInterceptor> 拦截器链，然后缓存。
这样，一个代理对象中，每个方法有属于自己的拦截器链。
    ↓
方法调用时，先按顺序执行拦截器链（前处理），再调用实际方法，再按逆序返回（经过每个拦截器的后处理）

但说实话，@PathVariable UUID id 这种场景下，加约束注解的必要性很低，原因：

类型转换本身已经是一道校验了——如果 URL 里传的字符串不是合法 UUID 格式，StringToUUIDConverter 转换阶段就会抛 MethodArgumentTypeMismatchException，根本走不到你方法体里，也走不到 @Validated 那一层
@PathVariable 对应的 URL 路径段，Spring MVC 路由匹配不到就是 404，不存在"传了个空字符串"的情况（除非你路径写了 /{id} 但用户访问 /，那是路由不匹配，不是校验问题）

所以 @Validated + 约束注解在 @PathVariable 场景下用得比较少，更多是用在：

@RequestParam 上的约束（比如分页参数 @RequestParam @Min(1) int page）
Service 层方法参数校验（@Validated 加在 Service 类上，配合 @NotNull、@Size 等）
@RequestBody 对象内部字段的级联校验（不过那个走的是 @Valid，是另一套机制，HandlerMethodArgumentResolver 里直接调用 Validator，不经过 AOP 代理）

你要是准备面试，这个点可能会被追问："@Valid 和 @Validated 有什么区别"——这俩机制完全不同：

@Valid（javax/jakarta.validation）：用在 @RequestBody 参数上，MVC 参数解析阶段直接调用校验，不走AOP
@Validated（Spring 自己的注解）：用在类/方法上，走AOP代理，能做分组校验（@Validated(Group.class)），@Valid 不支持分组
