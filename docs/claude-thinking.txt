关于为什么不建议用随机 UUID 做主键：因为聚簇索引按主键有序存储，随机主键会导致插入位置随机，频繁触发页分裂，降低缓冲池的局部性，造成写放大。而且二级索引的叶子节点存储主键值，UUID 占用 16 字节会增加索引大小。

应该用有序 ID 方案如 Snowflake、ULID 或 UUID v7，并在表设计中分离自增主键和业务唯一键。

当 SQL 突然变慢且 EXPLAIN 显示走了错误索引时，可能是统计信息过期或数据倾斜导致优化器成本估算失真。短期可用 FORCE INDEX 或优化器 hint 止血，长期需要调整索引设计或改写 SQL 逻辑，同时强调生产环境需要慢查询监控和发布前后的执行计划对比。

现在来讲 MVCC 在 InnoDB Repeatable Read 隔离级别下如何实现无锁读：每行记录有隐藏字段 trx_id 和 roll_pointer 指向 undo log 版本链。
快照读时事务建立 Read View 记录活跃事务列表和高低水位，沿版本链找到对当前事务可见的版本。RR 隔离级别下一个事务维持一个 Read View，而 RC 隔离级别每条语句都建立新的 Read View。当前读操作如 UPDATE 和 DELETE 则需要加锁。

RR 下当前读使用 next-key lock 同时锁定记录和间隙来防止幻读。所以 RR 的防幻读机制是快照读依靠 MVCC、当前读依靠 gap lock 的组合，混用两种读方式会产生幻象现象。
没有 MVCC 的话读写就会互斥，吞吐量会大幅下降。

长事务很危险因为 Read View 会钉住 undo 版本链导致无法清理，造成 undo 日志膨胀和历史版本遍历变慢，最终引发慢查询雪崩，同时 purge 进程也会滞后。需要监控 information_schema.innodb_trx 中的长事务，应用层要缩小事务边界。

在生产环境中很多互联网公司默认用 RC 隔离级别因为 gap lock 更少导致死锁和并发冲突减少，但 binlog 必须用 ROW 格式。RR 的价值在于单个事务内多次读取需要一致性的场景，比如对账或批处理快照。我的项目采用 RR 加显式 FOR UPDATE，关键的资金路径通过显式锁而不是依赖隔离级别来保证，隔离级别只是默认行为，重要路径需要明确声明意图。

现在考虑分布式事务的选型，首先要遵循的原则是优先重新划分边界让

系统不需要分布式事务。然后根据不同方案的特点来权衡：2PC/XA 提供强一致性但是同步阻塞、协调者是单点、吞吐量差、跨异构系统困难，只适合数据库内部或极少数场景。
TCC 通过业务层实现预留资源来达到一致性和低延迟，但业务侵入非常大，每个操作要写三个接口还要处理空回滚、悬挂、幂等这三大坑，适合资金类的短流程。

Saga 把长事务拆成本地事务序列加补偿，是最普及的方案，关键是理解补偿不是回滚——中间状态已经对外可见所以隔离性丢失了，补偿是真实的业务动作比如退款而不是删除支付记录。
在编排和协同之间选择时，超过三步的流程用编排会更清晰，否则逻辑散在各个服务里没人能看懂。我的项目选择了 Outbox 加事件最终一致性这个方案。

具体到我们的授权链路，额度扣减和交易记录放在同一个库的同一个事务里不拆开，但通知和统计这些下游操作走 Outbox——本质上是给每条服务边界标注清楚一致性要求。

关于 Saga 补偿失败的问题，补偿动作必须设计成幂等且可重试直到成功，也就是说最终一定能成功的操作比如记账冲正，不能是那种可能被拒绝的业务动作。如果真的失败了就进人工干预队列并报警。所以在选择补偿动作的时候就要考虑这一点。

选补偿方向时要挑"不会失败"的操作——加钱方向可以重试，但扣钱方向可能因为余额不足而失败。面试官可能会追问"那跨资源强一致做不到了？"，答案是单个资源内可以用数据库事务做强一致，但跨资源之间只能做最终一致，然后用对账来兜底。真的需要跨资源强一致时，优先考虑把资源合并到同一个库而不是引入 XA。

现在进入分库分表的话题——什么时候真的需要分库分表，以及 shard key 怎么选。首先要排除伪需求，单表几千万行在有好索引的情况下其实没问题，应该先做读写分离、数据归档或者加缓存。
真正的信号是写入吞吐超过了单机的能力。

shard key 的选择要按最高频的查询维度来，比如发卡系统就选 account_id 或 user_id，这样能保证单个账户的操作都落在一个分片上，本地事务就能保住一致性。跨分片的查询则走异构索引，把按商户维度的查询同步到 ES 或者读模型。分片算法可以用哈希或者范围分片，一致性哈希或者固定槽位的方案是先分出 1024 个逻辑槽映射到 N 个物理库，扩容时只需要迁槽不用重新哈希。分布式 ID 用 snowflake 要注意时钟回拨问题，或者用号段模式。扩容本质上就是一次小型的数据迁移，通过双写、CDC 加校验，最后切读流量。

对于热点账户打爆单分片的情况，需要先识别热点，然后要么单独隔离到 VIP 分片，要么对热点键再做一层拆分，比如子账户拆分加异步汇总。分片后的 count、分页、join 这些操作，
聚合可以走离线或读模型，深分页用 cursor 来做，设计时就要把哪些查询必须实时精确的需求清单化，通常会比想象中少很多。

现在看到一个关于线上 JVM 服务频繁 Full GC 或 OOM 的排查问题，先要看 GC 的形状再去抓证据。

首先判断是缓慢上涨后 OOM（内存泄漏）、瞬间打满（大对象或流量突增）还是 Full GC 频繁但不 OOM（老年代压力或元空间问题），通过 GC 日志和监控曲线的形状来判断。
然后用 jstat 看各代占用和 GC 频率，如果已经 OOM 就看 heap dump（提前配好 -XX:+HeapDumpOnOutOfMemoryError，事后用 jmap 会触发 STW 要谨慎），用 MAT 看支配树和 top 对象，或者用 jcmd 看类直方图做轻量级排查。常见的根因包括无界缓存或集合（比如 static map）、ThreadLocal 未 remove（线程池里线程复用）、大结果集查询没做分页等。

止血和根治要分开：滚动重启只是止血，必须先留下 dump 再重启，否则证据就没了。对于 G1 和 ZGC 的选择，要看停顿目标——G1 百毫秒级停顿、吞吐好、是默认选择；
ZGC 是亚毫秒停顿、适合大堆、吞吐略降、JDK17 以上生产可用。如果授权链路的 p99 延迟敏感可以考虑 ZGC，但要先确认停顿真的来自 GC 而不是锁或 IO，数据要先于选型。如果 GC 正常但 RSS 持续上涨，那就要看堆外的问题。

堆外内存可能来自 DirectByteBuffer、Netty、JNI、线程栈数量或 glibc malloc arena，用 Native Memory Tracking 来排查，这说明 JVM 内存不等于堆。

对于 Java 内存模型，volatile 解决的是可见性和有序性问题——它禁止重排序但不保证原子性，比如 i++ 还是需要 Atomic 或锁。经典场景是双检锁单例中用 volatile 防止半初始化对象逸出。synchronized 和 Lock 提供互斥加 happens-before 保证，而 CAS 操作是无锁的但有 ABA 问题和自旋开销。

ConcurrentHashMap 在 JDK8 后用 CAS 加节点级 synchronized，size 只是近似值，复合操作还是要用 compute 或 putIfAbsent 而不是 get 加 put。
虚拟线程改变了游戏规则——阻塞不再占用 OS 线程，让 thread-per-request 模式复活了，线程池的意义从池化线程变成限制并发的信号量。但要注意 synchronized 块内阻塞会 pin 住 carrier 线程（JDK21 有这个问题，24 版本改善了），ThreadLocal 容易滥用，而且数据库连接池这类底层资源仍然是硬上限——虚拟线程解决的主要是线程稀缺的问题。

回到我的项目，线程池大小是按下游容量算的，换虚拟线程这个约束不会变。如果升到 JDK21 用虚拟线程，收益是授权链路里那些外部风控调用等阻塞 IO 不再占线程，
同样容量下线程内存开销也降了。但风险和不变的地方是数据库连接池、行锁串行化、限流阈值都不会因为虚拟线程改变——当吞吐瓶颈在下游时，虚拟线程只是让排队变得更便宜。得先压测确认 pinning 的情况。

现在讲 API 版本演进，核心是兼容性铁律：加字段可以，但改语义、删字段、改类型都不行。调用方要按"容忍读"的方式编写，忽略未知字段。破坏性变更的做法是新版本端点或新字段并行，
经过迁移期，观测到旧版本流量归零才下线。版本策略可以用 URL 里的 /v1 或 header，但更重要的是组织纪律——契约先行用 OpenAPI，CI 里做 breaking change 检测。关于分页，offset 方式在深分页时是 O(offset) 的复杂度，而且数据插入会导致漂移（重复或跳过记录）；cursor 用 keyset 的方式（WHERE id > last_id ORDER BY id LIMIT n）是 O(1) 且稳定的，代价是...

不能跳页，排序键必须唯一且单调。错误设计方面要区分 4xx 和 5xx 的语义，错误体要结构化（code、message、trace_id），批量接口的部分失败用 207 或逐项结果。幂等 POST 用 Idempotency-Key 头。追问到内部微服务选 REST 还是 gRPC，gRPC 有强契约（proto）、双向流、性能优势，REST 则生态好、调试友好、网关友好，关键是 schema evolution 的纪律在 proto 里同样存在——field number 不复用、不改类型。JD 提到 gRPC 的话可以讲 proto 兼容规则。

密码存储要用 bcrypt、scrypt 或 argon2，自带 salt 且成本可调，不能用 MD5 或 SHA+salt 这种太快的方案容易被 GPU 爆破，加上应用层的 pepper 更安全，
登录接口还要限流和凭证填充检测。服务间密钥绝不能进代码库或镜像，用 Secrets Manager 或 KMS 配合 IAM 角色注入，设计密钥轮换时要双密钥并存过渡。这块也关联到 PCI 审计——卡数据环境的密钥管理是审计重点。

Session 和 JWT 各有权衡：session 有服务端状态能即时吊销但需要共享存储，JWT 无状态自包含但吊销困难只能靠短 TTL 加 refresh token 轮换和黑名单。
金融场景通常用短命 access token 配 refresh token，敏感操作还要二次认证。JWT 常见坑是允许 alg=none、密钥混用、把敏感数据放 payload（其实只是 base64 编码），还有不校验 aud 和 iss。如果 access token 泄漏了，短 TTL 能限制风险窗口，refresh token 轮换加重放检测能在旧 token 被重用时全家吊销，敏感操作的二次认证和设备指纹也能进一步止损。

核心思想是假设泄漏一定会发生，设计的是把爆炸半径控制住。

现在转向 S3 的深度展开，在第二个追问后插入详细的数据同步管道部分，讨论全量回填加增量 CDC 的工程细节，特别是一致性起点的问题——快照时刻的 binlog 位点要记录下来，
全量导出用一致性快照，增量从那个位点开始回放。

如果做不到一致快照，就先开 CDC 缓冲再跑全量，然后增量从缓冲回放并用幂等 upsert 覆盖。回填作业按主键区间分 chunk，每个 chunk 几千行，用进度表记录可以断点续跑，
还要限速避免打挂旧库，用 claimable job 并行。转换层的旧模型到新模型映射必须是纯函数加单测，建立"语义字典"逐字段记录来源、取值枚举、谁写的、空值含义，还要处理旧系统的枚举值。

CDC 的顺序性问题在于 binlog 按行有序但多表事务会被拆行，新平台按账户聚合时要处理同一事务的多行到达间隙，用账户级校验兜底而不是追求逐事务原子重放。
数据校验分三层：行数和主键集合对比抓丢行，内容 checksum 用分块 CRC/MD5 抓字段错，业务不变量校验最值钱能抓住两边都错得一致以外的语义错，校验也要增量化只校验变更过的账户。

流量复制有三种方式：网关镜像复制原始请求最真实但要处理认证和幂等副作用，事件订阅让旧系统落库后发事件新系统消费重算但延迟较大。

对于授权这类有外部副作用的路径必须保证 shadow 侧无副作用，外部调用全部打桩或录制回放，风控调用要用录旧系统的真实响应给新系统否则风控结果本身不确定造成假 diff。
diff 引擎先归一化掩掉时间戳随机 id 和顺序无关字段，然后结构化对比按决策结果分支路径和差异字段聚类，每类计数加样本留存，出口标准是连续 N 天关键类 diff 为零且已知类全部有结论。假 diff 的来源包括数据复制延迟，新系统读到旧数据时用事件里带的版本号对齐或 diff 时重查确认。

非确定性问题如时间随机浮点，以及灰度期间两边规则版本不同步都需要规则变更双投。比对也要比性能，shadow 侧记录延迟分布，新系统不光要答案一样还要 p99 达标。

路由维度按账户而不是按请求，同一账户的读写必须落同侧否则会出现用户可见的不一致。灰度组选择先从内部员工卡开始，然后低活跃账户，再一般账户，
大客户和特殊状态最后。路由实现用前置 facade 查路由表，路由表本身要高可用加本地缓存加变更推送，账户迁移瞬间要有短暂双侧拒写窗口来等待在途请求。

回切不是简单的开关而是完整的流程，需要旧系统数据仍然新鲜，回切动作包括路由表批量改回、验证旧系统恢复正常、新系统转回 shadow，必须提前演习过。
按路由侧打标的全套指标要有量化的前进停止回退标准写在 runbook 里。

明确了 SoT 后，需要逐格分析双写期的失败矩阵，比如新写成功但旧写失败时，虽然不影响用户，但要记录差异队列异步补写旧系统，此时回切能力会降级，差异队列长度成为衡量回切健康度的关键指标。

最危险的情况是新写失败但旧写成功，这种不一致必须通过调整顺序来避免——先同步写 SoT，只有 SoT 成功才异步写影子副本，这样影子写不在用户请求路径上，就不会出现这种矛盾的格子。

所以双写的最佳实践是"同步写 SoT + 异步复制到影子 + 差异对账"，本质上就是主从复制模型，风险可控，而不是同步双写两边都要成功的分布式事务。对于反向同步，
新系统可能表达旧系统无法表达的状态，所以迁移期间要冻结新功能上线，或给旧系统加兼容字段来支持。

对于长尾状态，需要逐一盘点分期中、争议中、督促延滞、解约清算、休眠卡等各类账户，每类决定是硬迁用状态机映射、双轨让存量旧系统跑完新开的进新系统、还是等待窗口让其自然终结。迁移时点要避开出账日和扣款日这些关键业务节点，选在账户账单周期内的安静段进行。

最优方案是按账户的账单周期错峰迁移，每个账户在自己周期的安静点（扣款完成后到下次出账前）进行迁移，这样能天然摊平系统负载。

对于大项目的里程碑设计，每个阶段都要独立产生价值：M1 是数据平台化让新库成为分析查询源、M2 是读切流获得性能收益、M3 是 shadow 运行发现旧系统的 bug 清单本身就有价值、M4 才是写切流。

团队结构上需要平台小队负责管道和路由基建，加上领域对对（每个能力域配一个懂旧系统的和一个建新系统的），把旧系统专家的时间当作项目的瓶颈资源来排程。

迁移期间的新需求要三分法处理：优先只进新系统，其次有限额度的双实现，原则上禁止只进旧系统除非特殊审批。给双实现设预算上限，超额就是砍需求的信号。最后要警惕"90% 完成停滞"的陷阱，因为最后 10% 往往是长尾状态和特殊客户的处理。

计划时要把长尾当作一等公民来估时，而不是当成"剩下的杂项"。

对于 CDC 复制延迟导致新系统读到旧数据的问题，需要三层防护：先把端到端复制延迟的 p99 做成 SLI 指标，超阈值自动降级读流量回旧系统；关键读操作在写发生后的短窗口内（比如 60 秒）强制路由回旧侧，用写事件驱动临时路由；产品层面在余额页展示"更新中"而不是错误数字。核心原则是复制延迟不能靠"希望它快"来管理，必须有可观测的度量和自动降级机制。

预算砍半、时间翻倍时，要检验切分质量——因为每个里程碑本身就有独立价值，已完成的阶段收益保留，剩余工作按"继续留在旧系统成本最高"的域优先迁移（维护费用、风险最大的部分），
其他域停在双轨稳定态并明确运维成本。

关键是让"半途状态"本身成为可长期运行的稳定态，这是从第一天就要保证的设计性质——最差的设计是"不做完就全无价值"。

现在转向 Linux 排障的话题。当服务器 load 飙高时，首先要理解 load average 不等于 CPU 使用率——它包括可运行的任务和处于不可中断睡眠状态的任务，所以 load 高可能是 IO 等待导致的。前五分钟的诊断从查看整体 CPU 使用情况开始，包括用户态、内核态、IO 等待和中断的占比，然后找出消耗资源最多的进程。

根据不同的指标高低来判断问题根源：用户态 CPU 高通常是应用计算或垃圾回收，需要看线程栈；内核态高可能是系统调用风暴或锁竞争；IO 等待高则需要查看磁盘 IO 情况和具体是哪个进程在等待；软中断高可能是网络问题。对于 Java 进程，可以通过线程级别的 top 找到热线程，把线程 ID 转换成十六进制后在 jstack 输出中查找对应的栈信息。还要检查磁盘空间、inode 使用、内存可用情况、网络连接状态分布和系统日志中的 OOM 或硬件错误。一个常见的误区是只看 CPU 百分比而忽视 IO 等待，这样会错误地认为服务器没问题。

关于容器环境下进程内存没超但被 OOM killer 杀掉的情况，这是因为 cgroup 的内存限额是从容器整体视角计算的，包括 RSS、page cache 等多个部分。JVM 的堆、堆外内存、线程栈和 metaspace 加起来可能超过了 cgroup 限额，所以要在 dmesg 或日志中查看 OOM kill 记录，并通过设置 -Xmx 只占限额的 60-75% 来留出缓冲空间。

文件描述符耗尽时，新的连接会报"Too many open files"错误，表现为服务还活着但拒绝新连接。可以通过对比进程的文件描述符数量和 ulimit 限制来确认，泄漏通常来自响应体没有关闭或连接池泄漏，监控文件描述符使用率是必要的。

现在进入 IO 模型的讨论。

阻塞 IO 是一连接一线程的模式，线程在读操作上睡眠；非阻塞 IO 需要轮询浪费 CPU；IO 多路复用让一个线程通过 select/poll/epoll 监听数千个文件描述符，只在就绪时处理。epoll 相比 select 的优势在于就绪列表是 O(1) 查询而不是每次全量扫描，也不需要每次都拷贝文件描述符集合。

Redis 能用单线程扛 10 万 QPS 的真相是每个命令都是微秒级的纯内存操作，单线程加 epoll 事件循环就足够了，而且避免了锁竞争和上下文切换开销。Redis 6.0 后把网络读写多线程化了，但命令执行仍然是单线程以保持无锁模型。单线程的代价是一个慢命令（比如 KEYS 或大 key 操作）会阻塞所有请求，这就是为什么生产环境禁用 KEYS 命令和要拆分大 key。

Tomcat 这类应用服务器采用 NIO 接入加业务线程池的架构，因为业务逻辑需要做阻塞的数据库或 HTTP 调用，延迟在毫秒到百毫秒级，必须靠线程数来堆并发。Netty 的做法是用事件循环组加 pipeline，业务中的慢操作必须从 EventLoop 中丢出去，否则整个连接组会卡死。

高性能组件的核心规律就是少拷贝、少切换、顺序 IO、批量化——这套思路在 Kafka 和 RocketMQ 的高吞吐设计中也体现得很清楚，顺序写加 page cache 加 sendfile 零拷贝能减少两次内存拷贝和两次上下文切换。

Bean 生命周期从实例化、属性注入、Aware 回调、BeanPostProcessor 前置、init 方法到后置处理器，AOP 代理就是在后置处理器阶段生成的，这解释了为什么代理能拦截方法调用。循环依赖通过三级缓存（原始 bean、半成品、代理 bean）来解决。

关于 REQUIRES_NEW 的实际场景，通常是需要无论外层事务成败都要保留的数据，比如审计日志或失败记录表，这样外层回滚时新事务的提交不会被影响。但要注意一个坑：如果外层持有某行的锁再开启新事务，新事务也要修改同一行就会产生死锁。

单例 Bean 不担心并发问题的原因是它们通常是无状态的，只存储依赖引用而不放可变字段。一旦在 Bean 中定义可变实例变量就会成为并发 bug，比如 SimpleDateFormat 作为成员变量就是常见的事故。如果确实需要有状态，要么用 ThreadLocal（记得清理），要么把状态放在方法内的局部变量里。

CAP 定理经常被讲错，正确的理解是网络分区（P）不可避免，真正的权衡是分区发生时选择一致性（拒绝服务）还是可用性（容忍不一致）。平时没有分区时的取舍其实是延迟和一致性的权衡，这就是 PACELC 定理。一致性存在光谱：从线性一致（像单机一样）到顺序/因果一致再到最终一致，对应的复制策略从同步复制、Quorum 读写到异步复制。在项目里的体现就是数据库主从架构中，关键路径（比如转账）走主库读保证强一致，而读模型可以走从库接受最终一致。

共识算法之所以难是因为异步网络环境下无法区分节点真的死了还是网络太慢，这就是 FLP 不可能性的直觉。Leader 选举容易出现脑裂，Raft 通过任期号加多数派投票加日志多数派复制才提交的方式来解决——多数派的特性保证任何两个多数派必然相交，旧 Leader 无法凑齐新的多数派。

关键是 fencing token 的概念：即使分布式锁服务本身没问题，客户端也可能因为 GC 停顿或网络延迟而拿着已过期的锁继续写数据。解决方案是锁服务发放单调递增的 token，存储层拒绝小于已见 token 的写操作。这说明互斥的最后防线必须在资源侧而不是锁侧，这也是为什么项目里的转账路径用数据库行锁或 lease（资源侧）而不是 Redis 锁（锁侧）。

对于业务的一致性选择，额度扣减是 CP 模式——数据库不可用宁可拒绝授权，因为授权的额度检查必须强一致；而通知、积分、读模型是 AP 模式——允许延迟和暂时不一致，通过重试和对账来收敛。设计的核心就是给每条数据标注它需要的一致性等级。

我在梳理一次服务间 HTTPS 调用的完整链路：从 DNS 解析、TCP 三次握手、TLS 握手到 HTTP 层的多路复用，每一步都有成本。冷连接需要 2-3 个 RTT 的建连延迟，这就是为什么必须用连接池和 keep-alive。超时不是一个而是三个——连接超时、读超时、连接池获取超时——而且总预算必须小于上游对我的超时，否则我还在重试上游已经放弃了。我项目里外部风控调用的 timeout 加降级策略就是这套预算法的实践。

关于 TIME_WAIT，几万个本身不致命，真正的症状是大量短连接在被创建和关闭，说明 keep-alive 没生效。风险在客户端侧会耗尽本地端口，根本解决办法是连接复用而不是打开 `tcp_tw_reuse`。另外要注意反向代理和客户端的 idle 超时错开，避免竞态。

对于客户端超时但服务端执行成功的情况，这不是 bug 而是分布式的本性——超时取消的是等待不是执行，客户端只能知道结果未知。正确的设计是用幂等键加重试，同一个键真成功了返回原结果，真失败了重新执行。

幂等不是锦上添花的优化，而是网络会超时这个物理事实逼出来的必选项。

现在看 load 高的问题。首先要纠正概念，load average 统计的是可运行加不可中断睡眠状态的任务数，所以 load 高不等于 CPU 高——CPU 可能空转，一堆线程卡在磁盘或 NFS 上，load 照样爆表。这个概念理清了，后面的排查才有方向。登录后前五分钟我会按顺序用工具链排查，先从 `top` 开始看三件事。

`top` 里看 load 三个值的趋势、CPU 分解（用户态、内核态、IO 等待、软中断），还有头部进程。然后根据 CPU 分解的具体数字分型排查：如果用户态高说明应用自己在烧，转向 Java 进程内定位；内核态高通常是系统调用或上下文切换风暴，用 `vmstat` 看 `cs` 列，常见根因是锁竞争、线程过多或频繁小 IO；IO 等待高就用 `iostat` 看设备利用率和响应时间，再用 `pidstat` 定位具体进程；软中断高说明网络流量或小包风暴。

对于 Java 进程，用 `top -H` 找烧 CPU 的线程，把线程号转十六进制，然后在 `jstack` 输出里按 `nid` 对号入座，那根线程的栈就能看出是 GC、死循环、正则回溯还是序列化风暴。同时顺手检查磁盘空间、inode、可用内存（注意别把 page cache 当泄漏）、连接状态分布和 `dmesg` 里的 OOM killer 或硬件报错。

OOM killer 看的是进程总内存而不是 JVM 堆，堆外还有 Metaspace、线程栈、DirectByteBuffer、JNI 和 glibc 碎片，容器场景常见的错误是 `-Xmx` 设得接近 memory limit 导致堆外没有余量。对策是让堆占 limit 的 60–75%，或用 `MaxRAMPercentage` 让 JVM 感知容器配额，被杀后从 `dmesg` 的 oom_kill 记录里能看到当时的内存账单。

文件描述符耗尽的现象很有欺骗性，进程活着、健康检查可能还通着，但新连接全部失败，像「半死」状态。fd 的消费者是网络连接、打开的文件和 pipe，泄漏源头常见是 HTTP 响应体没关或连接池配置错导致连接只借不还。排查方法是用 `ls /proc/<pid>/fd | wc -l` 对比 `ulimit -n`，看 fd 都是什么类型。

防御策略是把 fd 使用率进监控，用到 70% 就该报警，同时代码里走 try-with-resources 的编码纪律确保资源关闭。

IO 多路复用解决的是 BIO 的万连接万线程问题，通过让内核管理就绪状态，一个线程用 epoll_wait 看住几万个 fd，就绪了才去处理。Redis 单线程能扛十万 QPS 是因为命令都是纯内存操作微秒级，瓶颈在网络 IO 而不是 CPU，单线程加 epoll 事件循环就能喂饱，还白赚了免锁和天然原子性。Redis 6.0 的多线程也只是把网络读写并行化，命令执行仍单线程保留了无锁模型的核心优势。

但单线程的代价是一个慢命令会阻塞所有人，所以生产禁用 KEYS、大 key 要拆、删除用 UNLINK 异步化。Tomcat 要线程池是因为业务逻辑要做阻塞的 DB 和 HTTP 调用，毫秒到百毫秒级，这段时间线程必须干等，所以并发能力就等于线程数。这是两个模型的分水岭：处理是否阻塞决定了要不要用线程数堆并发。Netty 的 EventLoop 组也有同样的铁律，慢操作必须丢出 EventLoop 线程，否则同组所有连接一起卡。

Kafka 的高吞吐本质上是「顺序 IO 加零拷贝加批量化」的组合，分区日志只追加把随机写变顺序写，消费路径用 sendfile 零拷贝让数据从 page cache 直接进网卡，生产者攒批加压缩把每次操作摊到几百条消息上。高性能中间件翻来覆去就是四件事：少拷贝、少切换、顺序 IO、批量化。

现在看 Spring 的 @Transactional 失效问题，原理是 AOP 代理——Spring 给 bean 生成代理对象，事务的 begin/commit/rollback 织在代理方法的前后，所以失效场景归结为调用没有经过代理或异常没有到达代理。自调用是最高频的坑，this.methodB() 走的是原始对象不是代理，B 上的 @Transactional 完全不生效。方法不是 public 或是 final、static 也织不进去。异常类型不对是高频 bug，默认只对 RuntimeException 和 Error 回滚，checked 异常不回滚。异常被吞掉的话代理根本看不到，照常提交。

跨线程调用时事务上下文绑在 ThreadLocal 上，@Async 或自开线程里没有外面的事务。传播行为误配也容易出现一半提交一半回滚的情况。

Bean 生命周期里，AOP 代理是在后置处理器这一步把原始对象换掉的，循环依赖的三级缓存也是为了在这个流程里能提前暴露还没织完的引用。

REQUIRES_NEW 真正需要的场景是无论外层成败都必须留下的记录——审计日志、失败流水、重试任务这类，外层业务回滚了，这条记录也要活。

但有两个坑：自死锁问题，外层事务拿着某行锁，内层新事务又要锁同一行就会自己等自己；还有连接占用翻倍，同一线程同时占两个连接，连接池小并发高时会出现所有线程各拿一个连接都在等第二个的池死锁。所以 REQUIRES_NEW 要短、要少、别碰外层锁过的行。

Spring 单例 bean 被并发调用通常不担心线程安全，是因为约定 bean 无状态，字段里只有注入的依赖，它们也是无状态单例。

所有数据全在方法参数和局部变量里，栈上数据天然线程隔离。一旦有人往 bean 里放了可变实例字段就是并发 bug，经典事故是 SimpleDateFormat 成员变量内部有共享缓冲区，并发下日期解析出错乱值。需要状态时的出路是方法内局部变量、不可变对象或 ThreadLocal，但线程池环境必须 remove，否则值串到下一个请求。

现在来看 CAP 的真实版本——P 不是三选一里的可选项，网络分区是物理现实一定会发生，真正的选择只在分区发生时要 C 还是要 A。

平时没分区的日子里，取舍其实是延迟 vs 一致，同步复制慢但一致，异步复制快但有落后窗口，这就是 PACELC 对 CAP 的补全。一致性是光谱不是开关，从线性一致到因果一致再到最终一致，实现上对应不同的复制策略，quorum 可到强一致，异步复制只能最终一致。工程上一致性是按数据分级购买的，额度和交易走单点强一致，通知和读模型走最终一致，全系统一个档位既贵又慢。共识难的根本原因是异步网络里无法区分节点死了还是网络慢。

朴素的 leader election 会脑裂，Raft 通过任期号单调递增和多数派投票来解决，任何两个多数派必有交集，所以旧 leader 凑不齐多数派、新 leader 一定包含全部已提交日志。Fencing token 把理论落到事故上，即使锁服务完美，客户端拿到锁后可能因为 GC 停顿或网络卡住导致锁过期被别人拿走，它醒来不知道自己已经没锁了，继续写就会产生冲突。

解法是锁服务发单调递增的 token，资源侧拒绝 token 小于已见最大值的写入——互斥的最终防线必须在资源侧，这正是我项目用 DB 行锁和 DB lease 而不是 Redis 锁的理论根据。

额度与交易是 CP，授权的额度检查必须读到最新值，分区时选一致、弃可用；通知、积分、读模型缓存是 AP，组件故障不阻塞主链路，分区时选可用、容忍暂时不一致。真正的设计动作不是系统级的「选 CP 还是 AP」，而是给每类数据标注它买的是哪档一致性、故障时牺牲什么——把问题从二选一重构成分级采购，通常就是这题的最佳答案。

双峰分布（大部分 20ms 加一小撮 2s）和长尾渐变是完全不同的问题，双峰几乎总是撞上了某个离散事件，比如 GC、锁、重建或超时重试。

第二步是时间相关性分析，把慢请求的时间戳导出来，和所有周期性事件对齐——GC 日志、定时任务、缓存过期、连接池回收、部署时间、整点流量这些。我的习惯是先问尖刺有没有周期性，有周期就意味着有定时源头，一半的偶发问题在这一步就能破案。第三步是维度切分，把慢请求按实例、接口、账户、参数特征这些维度来分类。

现在要看这些慢请求是集中在单实例（宿主机或邻居问题）、单账户（热点行锁）、特定参数（某个分支的慢查询），还是完全随机分布（基础设施层问题），这决定了下一步往哪个方向挖。

第四步是请求级证据，trace 采样必须保留慢样本，用尾部采样而不是头部随机采样，否则低采样率下根本抓不到 p99 的样本。日志里用 traceId 贯穿整个链路，在关键阶段打点记录耗时，这样任何一条慢请求都能被解剖出时间花在哪一段。

最后是常见嫌疑人清单，按命中率排序，GC 停顿是首当其冲的。

现在我想把 W1 设定为这个综合开场题：一个需求从接到手到上线，你的完整流程是什么？这样能串联需求澄清、设计、拆解、开发、review、测试、发布、观测和复盘各个环节，每步说出关键动作和为什么要这样做。追问可以问哪一步最常被跳过、代价是什么——通常是设计和观测被忽略，跳过设计就得用代码试错，跳过观测功能上线了也没人知道效果怎样。W2 转向技术选型，讨论 boring tech、维度清单、单向门和双向门的区别、PoC 的时间盒限制，还有用 ADR 留下决策记录。

追问可以问有人提议用新技术栈重写时怎么评估，或者选型后来被证明是错的怎么办——这时候要看退出成本是否提前设计过、有没有通过抽象隔离来降低风险。W3 讲 git 和 PR 工作流，强调小 PR 的重要性，用 trunk-based 加短分支、原子 commit 配好 message 方便未来用 git blame 或 bisect 追溯，PR 要控制在 400 行以内，还可以用 stacked PR 来拆大功能。追问可以问怎么把一个大功能拆成小 PR——用 feature flag 做纵切或者先做准备性重构来"make the change easy"，还有 commit message 规范的实际价值。W4 开始问写代码之前做什么，设计文档怎么写才有效。

一页纸模板包括背景、目标、非目标、方案、失败场景推演、替代方案和否决理由，非目标本身很有价值，因为在纸上失败的成本远低于代码里失败。ADR 和设计文档有区别。追问可以问多大的改动需要文档——不是看行数，而是看不可逆性和影响面；如果团队没这文化，自己先写，用效果说话。W5 讲估算和交付管理，工程师低估的结构性原因是只估 happy path 的编码时间，忽略了集成、测试、review、等待、打断这些。要拆到两天以内的粒度，粒度是估算准确性的主决定因素。不确定性高的先做 spike，里程碑对外承诺要留 buffer，slip 的处理还没说完。

W6 讲高效开发的核心是反馈回路速度乘以打断保护，具体做法包括本地一键起环境、秒级测试、热重载，自动化所有重复操作，深度工作块配合异步沟通，减少在制品数量。

新入职的人可以通过先投资环境和构建来快速达到高效。W7 讨论 AI 辅助开发的真实收益和代价：优点是生成样板代码、测试用例枚举、学习陌生库、充当橡皮鸭和大代码库问答，但也带来幻觉 API 导致验证成本转移、生成代码膨胀增加审查负担、手写能力和调试直觉退化风险、审查疲劳降低警惕，以及团队技能差距扩大等问题。

对策包括核心逻辑保持手写、AI 产出要求最小改动、定期无 AI 练习和在 prompt 中明确项目约定。关于 junior 培养，担心他们用 AI 跳过挣扎期会缺乏调试直觉，所以让他们先自己思考再对比 AI，review 时追问原因。还需要考虑技术债的话题，虽然 T2 部分涵盖了，但用户提到范围包括技术问题、代码、项目管理和团队合作。

W8 可以问技术债管理，区分有意和无意、战术和战略的债务，用台账化和定期 review 来追踪，明确顺路修的边界，把债翻译成业务语言，通过 definition of done 防止新债产生。对于业务方不给还债时间的情况，可以把还债嵌进需求实现路径，用摩擦数据说话，大债通过下次需求顺路解决。

现在要确定这 8 个问题在整个框架中的位置——W 属于日常工程实践，应该插在 P（生产实战）之后、S（设计）之前，这样需要重新编号后续的章节。

还要检查与其他部分的交叉引用：T6 和 W7 涉及 AI 的质量与效率维度需要互相关联，B3 的 deadline 框架在 W5 中被引用，T2 和 W8 都涉及技术债，T5 的需求澄清与 W1 相关，P5/P6 在 W1 中被提及。此外要更新编号约定表、第一部分关于问题提问时机的说明、核对清单和 README。

现在开始写 W 部分的内容，每个问题保持实质性的深度，用结论先行的风格配合具体实践和反事实论证，自然地挂回项目背景。

这一节的核心是向面试官展示我是否是一个能持续交付的工程师——有自己的工作系统而不是凭惯性做事。每个回答都要包含具体做法、背后的原因，以及一个真实的反面教训。

对于需求从接收到上线的完整流程，我会强调几个关键环节：首先在澄清阶段用具体案例逼出精确需求，产出一页行为描述让需求方确认，因为写下来的分歧成本最低；其次根据影响面决定设计深度，小改动直接开发，但涉及数据模型或跨模块的必须先写设计文档，重点关注失败场景；然后拆解任务到两天以内的粒度，识别有回滚依赖的步骤；最后开发阶段用短分支小 PR 的方式推进。

发布前自己先 self-review 一遍 diff 能拦住一半的低级问题，发布后用金丝雀加指标阈值的方式灰度，并主动盯 30 分钟；收尾时不只验证"没报错"，而是看业务指标是否真的动了，清理发布型 flag，顺手更新文档。这个流程最常被砍掉的是澄清和收尾两步——跳过澄清就会用一周的返工代替一小时的对话。

跳过观测验证意味着功能上线了但没人知道它有没有真正起效。我见过代价最大的是设计文档里漏掉"非目标"这一段——不明确说不做什么，开发中途就会被 scope creep 吃掉，每个人都往里塞"顺便做了吧"，两周的活变成六周。非目标是设计文档里唯一防守性的段落，给了工程师说"不做"的书面依据。

技术选型我用四个维度打分，权重根据团队现状调整。首先看问题匹配度——这个工具解决的真的是我们的问题吗，还是为了用工具而发明问题。其次是总拥有成本，包括学习曲线、运维负担和升级路径，特别是引入新的存储系统或消息队列这类运行时组件，运维成本远大于库和框架。第三是生态与生命周期，看社区活跃度和大厂背书，我倾向于选择无聊的成熟技术，把创新预算花在业务差异化上。最后是退出成本——

这个决策是可逆的还是不可逆的。

可逆决策（比如换个 JSON 库）可以快速试错，但不可逆决策（选主数据库、定事件 schema）值得花十倍时间谨慎评估，决策的谨慎度应该和不可逆性成正比。执行上我限制候选方案不超过三个，用真实数据和负载做时间盒 PoC，最后写成 ADR 文档记录背景、选项、决定和被否决的理由，这样半年后新人问为什么选 X 不选 Y，答案在仓库里而不是在离职同事脑子里。

我的项目就是这样选的——用 MySQL 行锁而不是 Redis 锁做资金互斥，选 outbox 而不是 Kafka 事务，每个决定都带着否决理由在文档里。

当同事提议用新框架重写某个模块时，我会把讨论从审美拉回成本：先量化现状的痛点（bug 率、改动周期），确认新方案是否真的解决这些痛而不是别的问题，评估迁移成本和风险，最后看有没有渐进路径——能用 strangler 模式就不推倒重来。多数重写冲动在这一步就会被打消。

但也不能一刀切当守旧派，如果新技术确实命中真痛点，就在小范围试点并收集数据。

如果选型后来被证明错了，关键是设计时就该给依赖留抽象层——repository 接口后面藏 ORM，事件发布藏在 port 后面——这样退出成本是设计出来的。真要换的话，承认沉没成本不是理由，算的是从今天起继续用还是迁移的前瞻成本，然后用双跑灰度的方式来迁移。

最重要的是把教训写回 ADR，记录当初哪个假设错了，选型能力的成长全在这些复盘里。

我的 git 工作流是主干开发加短命分支加小 PR。分支活不过几天，因为长命分支的合并冲突和集成风险随时间超线性增长，用 feature flag 让未完成的功能也能安全合入主干。PR 要控制在 400 行以下，研究和经验都指向 review 质量在几百行后断崖下跌，大 PR 得到的不是真正的 review 而是走过场的批准。

小 PR 的复利效应很明显：review 快就能合入快，冲突少，回滚粒度细，bisect 也好用。

原子 commit 加有信息量的 message 同样关键——一个 commit 一个逻辑变更，message 讲为什么而不是做了什么，因为 diff 已经说了做了什么。这不是代码洁癖，而是实用考虑，`git blame` 和 `git log` 是最常用的考古工具，半年后面对一行怪代码，blame 出来的 message 写着"兼容 X 系统的日期格式 bug"和写着"fix"，完全是两个世界。我准备项目时坚持每次提交都带完整 message，就是给自己的考古留路。git bisect 在回归 bug 时特别有用，能快速定位问题引入的具体提交。

分支内 rebase 保持线性好读，但共享分支绝不 force push；用 `git revert` 生成反向 commit 保留历史可追溯性，而不是删历史。

现在的问题是大功能怎么拆成小 PR，答案是两个刀法：先做准备性重构让改动变容易，再做容易的改动。

第一批 PR 全是行为不变的重构——抽接口、挪代码，测试证明行为没变，最后才是真功能，每个 PR 都能独立 review 和回滚。第二个刀法是纵切加 flag，按"走通一条最窄的端到端路径"来切而不是按层切，逐步放开 flag。那种"一个 PR 3000 行因为功能不可拆"的说法几乎总是假的，其实是没花拆的功夫。

现在进入第四周的话题：写代码之前做什么，设计文档怎么写才不流于形式。核心原则是写文档是为了先在纸上失败。

在纸上改一个方案十分钟，代码里改一个方案就得花一周。判断什么规模需要文档看两个轴：不可逆性和影响面。改数据模型、事件 schema、对外 API 这些必须写，因为 E7、E9 的兼容性问题都源于此；跨模块、跨团队的改动也必须写。代码行数其实无关——改一行隔离级别的代码可能比一千行 CRUD 更需要文档。一页纸模板就够了，多了没人读：背景与目标用新人能看懂的语言讲清楚，非目标明确说不做什么防止 scope creep，方案部分画出数据模型和关键流程图。

失败场景推演很关键，每个环节都要想"进程在这里挂了会怎样"，这段逼出来的设计修正比评审会议还多。把被否决的替代方案列出来，一句话方案加一句话否决理由，能提前消灭评审会上 80% 的"为什么不用 X"。最后讲风险与迁移，怎么灰度上线、怎么回滚。评审的正确姿势是文档提前 24 小时发出去，会上现读就是浪费所有人时间，关键 reviewer 要提前一对一过一遍。

团队没有文档文化的话，不要开口号会，用行为示范更有效——自己的下一个中型改动写一页纸发出去，第一次可能没人看，但当它在评审里提前挡掉一个设计错误、或三个月后有人靠它十分钟看懂模块时，价值自己会说话。把门槛降到最低，提供模板、把文档链接加进 PR 模板的可选项。自上而下的文化靠强制，自下而上的文化靠"被看见的好处"，后者慢但牢。

工期低估是结构性问题，不是态度问题。我们默认估的是编码 happy path 的时间，但真实交付还要加上测试、review 等待、集成调试、环境问题、会议打断和需求微调。

编码往往不到总时间的一半。还有计划谬误，人对自己这次总是乐观，对别人或上次的教训不敏感。加上不确定性不对称，顺利只能省 20%，踩坑能翻 3 倍，所以估算分布是长尾的，而我们报的是众数。

我的对策是拆到两天粒度再估，粒度是估算准确性的第一决定因素。

拆不动的部分就是不确定性所在，先做时间盒 spike，一天内产出"知道怎么做了"就够。用历史数据校准，看上个类似任务实际花了多久，个人系数比任何方法论都准。对外承诺加缓冲并分级，对内计划紧、对外承诺松，milestone 用日期区间而不是单点日期。进度落后的第一时间暴露，估算错不是失职，隐瞒才是。

明知做不到的 deadline 被压下来，关键是把代价具体化让决策者签字。日常预防的两条路：争取估算参与权，deadline 应该是业务需要和工程估算的谈判结果，而不是单方面通知；平时积累估算信用，估算历史越准，谈判时数字越有分量，信用是复利资产。

高效开发的核心是反馈回路。

我的做法包括本地环境一键启动，新人从 clone 到跑通不超过 30 分钟；测试分层求快，单测秒级、集成测试选择性运行，因为测试慢的直接后果是大家不跑测试了；热重载和增量构建，构建超过几分钟就值得投资，因为它每天被乘以百次；自动化所有重复三次以上的操作。

对于专注保护，我强调深度工作块的重要性——大块不被打断的时间产出远高于碎片时间之和，因为上下文重建一次需要 20 分钟，一天被打断 6 次就是半天没了；采用异步优先沟通，非紧急问题走文字批处理；严格限制 WIP，同时只推进一件主要事，因为三件事各完成 60% 的交付价值是零，一件 100% 两件 0% 至少交付了一件。

新入职团队的快速高效之道是头两周把时间投在"环境和回路"而不是急着产出——跑通构建测试部署全链路、读懂 CI、配好调试环境，这些投资之后每天都在付息。同时用认知地图的方法追一条主链路并写下来让老人挑错。第一个任务选小而完整的，目的是把发布全流程走一遍——新人的首要任务不是证明聪明，是打通自己的交付管道。

现在转向 AI 辅助开发这个话题，想探讨它真正的提效点在哪里，以及带来了什么新问题。

按实际体感来看，AI 最大的价值在于处理样板和胶水代码——mapper、DTO、配置这类机械劳动几乎可以归零。其次是测试用例的枚举，AI 穷举边界 case 比人更有耐心，虽然后续还是要我来裁剪断言。陌生领域的考古工作也收益很大，比如读不熟的库源码、解释遗留代码、生成文档骨架，这块的效率倍数最明显。另外把设计讲给 AI 听让它来攻击，比对着墙讲要有效得多。在大 codebase 里做检索问答也很有帮助，快速定位"哪里处理了 X"这类问题。

但这也带来了新的问题。验证成本转移到了我这边——生成快了，但幻觉 API、过时用法、似是而非的并发代码需要我来验证，总时间省多少取决于我的验证效率，这也解释了为什么资深者提效更多。代码膨胀也是个问题，AI 偏爱防御性样板和过度抽象，不加约束的话代码量和 review 负担会同步上涨，我的对策是在 prompt 里强制"最小 diff"并把项目约定喂进上下文。还有审查疲劳的问题，大量"看起来都对"的代码容易让人警惕性下降。

我也在意技能维护成本——调试直觉和手写能力是靠挣扎积累的，全托管给 AI 会退化，所以我刻意保持核心逻辑手写或逐行重写 AI 稿，定期不带 AI 做题保持手感。最后是团队方差放大的问题，会验证的人效率翻倍，不会验证的人反而产出负价值代码更快了——AI 放大的是判断力差距而不是抹平。

归根结底，AI 把瓶颈从"写"移到了"判断"，受益最大的是本身判断力强的人。

关于 junior 培养的风险，我承认挣扎期确实是调试直觉长出来的地方。我带人时的调整是先让他们自己写思路再对比 AI 的，差异点就是学习点；review 时多问"为什么这么写"，讲不出来的就按规则打回；还给 junior 的任务里保留一部分"禁 AI 区"，比如手写并发原语或手查一次生产问题。这就像计算器没有消灭心算教育，只是让教育更聚焦于"什么时候该按什么"——AI 时代的教育也是这个逻辑。

现在来看技术债的管理。我觉得得先分类再谈，不分类的讨论都是各说各话。有意的战术债是合法工具，但必须记录——欠条不记债就变成惊喜了，我用 TODO 注释关联 ticket 或直接进债务台账。无意的债靠 review 和事后发现，发现即记录。还有腐化债，当年正确但世界变了，比如依赖过期或模式不再适配——这个需要定期审视。

管理的核心是给债务标注"利息"——这个债每月吃掉我们多少摩擦时间、bug 率或招人劝退。没有利息的债可能根本不用还，丑但稳定的代码还它反而是伪工作。还债走两条腿：小债顺路修，但要小，PR 里混大重构会毁掉 review；大债打包进需求，"做这个功能正好要动那个模块，加 20% 工时顺路重构"比单独立项容易批十倍。

关键是用摩擦数据而不是道德词汇来谈判——"烂代码"说服不了人，数字可以。控制新增债的方法是 definition of done 含测试和文档，review 挡住"临时方案不留欠条"。

业务方永远不给还债时间的真正解法不是申请专门的还债时间——那样永远输——而是把还债嵌进需求实现本身。
